7.1 Operands and Data Processing

Overview

Much of today was explaining design decisions from the CI Lab assignment. We learned more about the ARM architecture, and then about different operand types and inputs.

Notes

Arm's Memory Model

There are 2^64 places we can put things, but in ARM, we define two different twists to keep to things

  • Normal: Used for most reading and writing
  • Pointer subsets which point to something else, DEVICES!
  • Whatever I'm typing on this keyboard is interpreted as coming from a memory location. This way I can read off the input of a device through specific pointer
How do I find out what pointers are to the devices?

REMEMBER instructions must be word-aligned
Data alignment rules are complicated
For my ARM instructions, they are always little-endian.
Endianness of regular memory is configurable at EL1 or higher. Always assume little-endian unless told otherwise.

All memory mapped peripherals should also be little endian! To accommodate this, we get drivers.

Instruction Processing

Program evolution should always map internally to valid states. The state of a computer is contained within the processes of memory and the processor itself.


Notice how the data memory is in a separate spot from the instruction memory like the harvard architectureharvard architecture.

  1. Fetch the instruction from the location pointed to my the program counter
  2. Decode the instruction word (The lexer in CI Lab did this first step of pulling information in)
    1. Operation, operands, results
      • The instructions are like verb noun1 noun2
    2. Read off the operands from the registers
  3. Execute the operation determined off by this
  4. Access the memory
  5. Write the results back to the register
The reason that 5 is after 4 is because what I dump back might be contingent on what I read off of the memory.

After all this, you have to move to the next instruction!
6. Update PC with the value we need to assign to it (think of branching for when this might not be consecutive incrementing)
What we learn across the stages persists as we step.

Sources and Destinations

The form of an instruction is
IW = (Operation, Operands, result locations)
What are possible operands?

  • Variables I change, the registers/pointers, or even a straight up constant.
    • You'd do that when you want to increment a pointer by a known amount.
  • The results are placed in a register or groups of memory cells

Immediate Operands

A specification used for the constants within the instruction itself. They are also like the hex notation of #0x to tell us how to interpret the number (For assembly). Some compilers now don't need the #, but it's less ambiguous to place it anyway.

The interpretation is known and persistent across program states. These were used a lot in CI-lab!

I am terribly caffeinated.

Register Operands

The information that I want from is sitting in my registers. You call then using either an X or a W.

  • W uses the lower 32 bits only while the upper 32 bits are cleared to zero
    XZR,WZR is the zero registers
    SP,WSP is the current stack pointer.
  • The regular SP is 64bit and is not notated by X
