Huge Design Space

The header should be a fixed length header. That way we know exactly where the size parameter sits. The size is 16 bytes back from the pointer in this case. Then what comes right before our pointer is the pointer to the next free block.

The padding makes sure that the address that we have is a multiple of 16 bytes, always. We will not go through the reasons in here, but it's to do with keeping memory allocation efficiency (Is it related to memory buffers?).

Keep things 16 bytes aligned

It's worth it to waste some space for this sake

Free blocks also need to have size and padding. If I'm implementing a doubly linked list, no one is using the memory inside the payload. This doubly linked list becomes only doubly linked when it's not in use.

Free Block Selection

A circular linked list might make things easier

  • First Fit- When you pick the first possible place the memory can fit.
    • Pro: Time fast
    • Con: Biases your unusable small spaces near the beginning of the memory
  • Next Fit- First time start from the head, find something for the first request, then when the next request comes in start from where you stopped. Implement via circular linked list
    • Pro: Tiny holes are spread out across the memory
    • Con: Not sure…
  • Best Fit- Pick the block over all the data which best sizes the requested memory
    • Pro: Obvious heuristic, minimizes unused allocated memory
    • Con: makes really small unusable memory chunks
    • Causes internal fragmentation (fragmentation inside the space given internal to you)
  • Worst Fit- Pick the worst fitting block and allocate it
    • Pro: Makes larger usable chunks for blocks
    • Con: Can be more difficult for splitting? Didn't catch this
    • Causes external fragmentation (fragmentation in the blocks we can allocate)

Free Block Coallescing

Going from an allocated block to a free block, which means we have to update our datastructure. If the block being free is in the middle of two allocated block, there's no need for them to absorb each other.

We can also use BOUNDARY TAGS!
We can make coalescing faster if we give each block it's metadata at the header and the footer. Then the current block can check if the previous one needs to be freed by subtracting 16 bits from it's current pointer.
