Exceptions in PIPE

There are 3 internally generated exceptions, statuses which propagate down the stages.

  • Stat==HLT a halt instruction generated in fetch. Program completed!
    • You need everything else to finish before you're allowed to stop! because we're a pipeline machine, we can't stop until we're at the end of the line
  • Stat == INS imem_err is asserted, or an instruction with an invalid opcode. Generated in F. Memory error takes pace in the memory step
    • Writeback can't get that information, but when a problem is detected in a particular stage, people ahead have to run to completion because they also can have their own problems which take priority.
    • Exceptions do propagate back down the line
  • Stat == ADR dmem_err

State updates take place in XMW. They occur early in the pipeline.
If the problem occured in writeback, X and M are not allowed to make a state update. Remember we have to match what we expect from SSE. The effect of X and M are not allowed to be seen.

Priority is to the earliest excepting instruction.
If a problem occurred in W while we're in M, then the M stage might try to update the memory when we're not allowed to. The execute stage might try to update a register when we're just not allowed.

If we get a mispredicted branch, then in the branch there is a error, we should squash the new errored instruction loaded into memory
