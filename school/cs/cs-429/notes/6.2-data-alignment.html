6.2 Data Alignment

Overview

Today we covered how objects are placed in memory and how that behaves.

Notes

The Problem

It is possible that we run across a problem where we request a section of memory, it gets broken in half! The microprocessor ends up having to request both chunks.
Diagram of computer memory with stacked bytes. One charecter variable split half and half across two different bytes, despite being adjacent, now requires two accesses!
Notice that char a in the diagram is still contiguous! But now I'm still across 2 bytes

In the above case if I were to split one datatype across bytes, I'd need to for one char access memory twice. Either I spend time cleaning up my memory so that this doesn't happen, or I increase the I/O bus bandwidth to make more

We need to make sure that our objects are residing in well defined spots

Motivation

Basic datatypes have at runtime known sizes in memory, but structs don't!

AArch64 requires 4 byte alignment for instructions

Definition

For any object had for a particular type, then those types have to be on boundaries of a certain size as a multiple of byte addresses.
Something is aligned if

The address must be a multiple of the size.
A character can go anywhere, a short has to exist at only even registers, a character can go anywhere.

Did we need to accommodate that on the MM assignment?

Nope actual writing to memory is below our scope onto the OS. We just allocate

An aggregate object of a derived data type T (an array, struct, etc) is aligned IFF

  • Every sub-object is recursively aligned. Every sub-type must follow it's respective rules.
Does this mean that the order that I define datatypes in a struct can change the amount of space?

Like if I defined the short first, and then an int, would I reduce the space of the struct by changing the order?
YEP! Switching around the types in a struct can reduce it's memory size.

Declaring variables in an accommodating order will reduce their memory usage

And there's also the array rule, every element of an object y of type "Array of T" is also aligned.

Does that mean the SIZE of a struct/abstracted datatype, then is the alignment requirement a multiple of that?

NO! That only applies for primitives. We align abstracted datatypes based on the primitive type containing it which requires the most memory

For every object, there are two decisions which have to be made. What the address is, and what the size of the object in.

  • Basic types only have to decide the address
  • With complex types we have to also determine the size!
    Derived types demand this analysis on the padding.
