1.9 Smart Pointers

Overview

Today we're talking about how we might handle wanting to wrap our references to an object with more intelligence! Maintaining exclusive access to the door, sharing access to a door but keeping track of the size of the line, etc.

Smart Pointers
Diagram of the two-part future implementation

The original future object hides the FutureImpl from me the user. i can't get my hands on the pointer itself.
The destructor will subtract one each time, until the reference is zero, then we kill FutureImpl.

This is a concept we can apply to anything! All we need to do is track the lightweight pointers (smart pointers)

  • In python, every pointer is a smart pointer

There are two kinds of smart pointers

Unique Ref

This is just a pointer which points to something, and it promises that it is the only one which points to that data-structure.

  • This is useful because we don't have to worry about sharing! If I'm accessing it, no one else can
    We write it as UniqueRef<?> (the type doesn't matter). 
1
UniqueRef<Person> p{new Person()};
2
// makes a variable called p, the unique reference of some type is to p
3
auto q = p;
4
f(p);

What do we do in the case of q? If we want to guarantee that we have the only pointer to this object, we can make it illegal to attempt to copy it, or if you make a copy of it we revoke the original permission

What hapens if in the constructor for UniqueRef I pass in a pointer already? That external pointer isn't bound by our feeble rules

That's a design flaw! No good programmer should ever do this
A common pattern is

1
auto p = UniquePtr<int>::make(10)

This way we can set the initial value while still never getting our hands on the pointer

We also have a copy constructor. The argument it takes is a reference to an object that exists and is not null.
When we're transferring permissions across the UniqueRef, we don't need to worry about the time it takes to transfer ownership.

Strong Ref

A strong ref is an object which keeps track of a reference count. The object itself doesn't know it's wrapped in a strong reference
Diagram of the inner structure of a StrongRef

Our intermediate object holds our reference information, that way our smart pointers don't need to worry about all keeping track of each other. There's another implementation which is a bit more efficient
Intermediate Object Implementation

To implement this, we have some object which sits in the middle

When I make a copy of a UniqueRef in a StrongRef, I lose the uniqueness criteria in it. We do it by just carving out the guts.
We say that the StrongRef itself is actually NOT thread safe. It doesn't mean we can't put a lock on a struct to keep it safe. All we're saying is that we can't save it's internal state by itself.
The intermediate middle object has to be thread safe, as we know we have multiple copies of StrongRef pointing to this.

  • To make middle thread safe, we just need to make the counter atomic. it's what gets mutated the most frequently!

It's not our job to project the final object we point to. If they make it thread safe it will be! If the creator doesn't, then it won't. Whoever implemented the object is the one protecting it.

Defensive programing via ASSERT(...) is very important

If I want to have stronger oriented OOPs, we should make the Middle class have add and drop methods which return the final value, then the delete operator only needs to worry about deleting.
C++ has friend classes where you can set some classes to be allowed to interact with your class's private variables.

Weak Pointer
Diagram of our linked island problem

What do we do about this closed loop now? Well these two points in memory are now pointing to each other. These two objects can't be reached from the outside, with no purpose, but keeping each other alive.

This is an isolated island! Cycles of strong references will cause these isolated lines. The price we pay is leaked memory.

How do we fix this? We make it illegal for cycles in our access graph. We're going to make it a programming error which is undetectable by the compiler.

The price we pay is that whenever we access a WeakRef, we have to ask it if it's valid. WeakRef does not override the arrow operator.

We can think in terms of the middle class
Diagram where we distinguish weak and strong refs

A strong counter of 1 meaning that we're still connected to our larger graph of access. A weak counter means that someone connected to me is someone I connect to
How many strong references that keep it alive enable it's existence.
How many weak references who are tethered have the responsibility to release the remaining middle. We preserve the decayed middle only to tell us if a former object still exists, and it saying that the number of strong counts is zero tells us it's gone.
