Bounded Buffer

If you try to put somethign and the queue is full, you block into the lock for it. You get woken up by whoever makes the queue not full.

Diagram of the Bounded Bugger

In the bounded bugger, it gets more complex. Both methods can make a block. Block when you do a get and the queue is empty, or you do a block and the queue is full

All the complexity and difficulty is in the synchronization logic. The DS itself is something you need uniquely for everyone. The synchronization logic is reusable!!

There are two ways of doing this are very popular for good reason. One is a semaphore

CountingSemaphores

This is essentially an unsigned counter
Diagram of the Semaphore

this is wide enough so we can guarantee it will never overflow. You can never ask for the value of the counter, just change if it goes up or down.
We consider it safe to use a 64 bit counter, since 2^64 cycles is nor reachable within a century. Overflow will never happen. To prevent it from underflowing, we block. We protect it from underflowing by preventing down from succeeding. We wake up when someone says up.

Condition Synch Primitive

A Semaphore is a structure which has memory, it's stateful. If you add 1 to the counter you'll add it regardless of waiting people. it has a state which is remembered. Because it's stateful, we had to do the propagation of the state information down the chain. Can we have a synch primitive which is stateless.
When all we have is a queue,, we refer to that as a condition variable. It's a queue of threads which are blocked. The variables doesn't care about the why, they just are. It's a list of all the threads waiting on something.

Diagram of our primitive

Notify says to wake up at least 1 thread, notify all says to do all of them. If we had that (since it's a variable/type). There's no state if someone does a notify on it, it'll wake up everyone waiting. There's nothing inside me which says that I've been notified. It's stateless. Very little state needs to be implemented.

There is some lock which has to be associated with our setup. Either it's embedded in the code inmplicitly, or you pass it with arguments.
Attatching a condition with a lock, it becomes super useful.
Wait says "add me to the queue, then release the lock"

  • When you call wait, you must be holding the lock
