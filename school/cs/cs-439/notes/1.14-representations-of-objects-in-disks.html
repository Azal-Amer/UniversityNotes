Block Groups

We have a chunk of memory for the iNodes called iTable, ext2 made the decision that the bitmaps for the iTable and blocks would be 1 block
Diagram of our Block Groups

We're motivated to keep files in singular block groups. Then we know that the blocks for a given file are physically close to each other, so the head doesn't need to do a lot of movement.
Then the total iTable is assembled by the concatenation of all the iTables.

This is just a performance optimization to promise adjacency in blocks

Now because we have all these block groups, we need another setup. We acll it the Group Descriptor table, and each entry in it describes a singular block group!
The superblock starts us off, then a sequence of rules, then our GDT and now we can manage our whole system!

Buffer Cache

Knowing disks are existing, we reserve a portion of our memory (like a hashmap in memory) which hashes a block number to the contents of that block. It just remembers the result of read_block for a device. The key thing si the block number and disk id.

When we try to read from the disk, we actually go to the point in memory
"I want block #7 from disk #3, is that in the buffer cache" i
f it is, then we take it from there.
How do we know it's most recent?
Well we use software to make sure that everyone goes straight to the buffer cache first! If we don't find it, we bring it in.

Naturally it brings the working set from disk to bring it into memory/ We're motivated to use as much memory as possible for the buffer cache to hold all of working memory, but IRL we use like 20%.

The device interface basically says that no one ever hits the disk directly, they always go to the buffer cache. We can use an LRU[1] policy!
This makes a HUGE performance boost.
